2 Маршрут: Выкатка фичи

Полный путь от бизнес-требования до прода. На каждом этапе - автоматизация и SLA.

1
Бизнес-требование 1 рабочий день
Product Owner пишет в чат (голосовое/текст). Delivery Manager ловит и фиксирует в Sheets + Трекер.
/morning: напомнит если требование не зафиксировано Tracker: create_issue
2
Детализация карточки 2 рабочих дня
Delivery Manager заполняет: что сделать, для кого, зачем (бизнес-цель!), критерии готовности. Только бизнес-логика - без JSON и методов.
/morning: проверяет есть ли бизнес-цель в карточке Sheets: проверка заполненности колонок
3
Тройная верификация (Стендап) Ближайший понедельник
Delivery Manager называет номер задачи → бизнес-цель → что делаем. Три подтверждения:
Product Owner: бизнес-цель верна? Риски?   Tech Lead: понятно как делать? Срок?   Аналитик: оформление ок?
/morning: проверяет готовность карточки к стендапу Tracker: update_issue (статус → "Согласовано")
4
Задача → Tech Lead Оценка: 1д мелкие / 1нед глобальные
После тройного ок. Tech Lead владеет сроком и доставкой до прода. Сам ставит задачу Frontend Dev если нужен фронт.
Tracker: move_issue → "В работе", назначить assignee Sheets: обновить статус + дедлайн
5
Разработка 2-5 дней (мелкие фичи)
Tech Lead (бек) + Frontend Dev (фронт) работают параллельно. Катим сразу, релизов не ждём.
/morning: сверяет дедлайн в Sheets с Tracker, алертит при расхождении Tracker: search_issues (статус "В работе", просрочка)
6
Разработка завершена → Автотесты Результат за 1 час
Tech Lead переводит в "Готово к тесту". Клод подсвечивает фичи, готовые к тестированию, и предлагает DM запустить тесты:
- Фронт: формы, точки входа, редиректы (Selenium)
- Бэк: API, постбеки, расчёт комиссии
Результат → комментарий в задачу Tracker + уведомление DM
Автотесты: Selenium фронт + бэк Tracker: list_transitions → move_issue Telegram: уведомление DM о результате
7
Тест DEV (ручной) 1 рабочий день
Delivery Manager тестирует на DEV. Мини-отчёт: что проверил, результат. Если автотесты прошли - фокус на бизнес-логику.
Tracker: add_worklog + комментарий с результатом
8
Деплой на PROD 1 рабочий день
Tech Lead деплоит на прод. После деплоя - уведомление DM для тестирования на проде.
Telegram: уведомить DM "задеплоено, проверяй" Tracker: move_issue → "На проде"
9
Тест PROD + Готово 1 рабочий день
Delivery Manager тестирует на PROD. Мини-отчёт. Дедлайн = этот момент, не "код написан". Фича работает на проде.
Sheets: статус → "Готово", дата закрытия /morning: убирает из плана, включает мониторинг
Оценка выкатки: 2-5 дней (мелкие фичи), до 1 недели (глобальные фичи / крупные интеграции). Бэк и фронт работают параллельно, катим сразу. Блокеры (договор, партнёр в отпуске, ожидание МФО) пробивают любой срок.

1. Откуда будет трафик? Ваше приложение, сайт, СМС-рассылка, отказная страница - или сразу из нескольких источников?

2. Формат интеграции. Тут два варианта:

5 /morning - движок системы

Центральный движок автоматизации. Не просто текст - полноценный цикл: читает → сравнивает → действует.

/morning

Полный цикл работы

1
Чтение источников (параллельно)
Sheets: Партнеры, Фичи, Интеграции Tracker: задачи в статусе "В работе", "Готово к тесту" Telegram: get_mentions за последние 12ч Кеш: предыдущее состояние статусов
2
Сравнение с кешем
Текущие статусы vs вчерашние → выявление изменений:
- Какие партнёры сменили статус?
- Какие фичи перешли в "Готово к тесту"?
- Какие дедлайны прошли?
- Какие колонки опустели или заполнились?
3
Сверка Sheets ↔ Tracker
Статус фичи в Sheets = "В работе", а в Tracker = "Закрыто" → алерт. Sheets не обновлён.
Статус в Tracker = "Готово к тесту" → триггер автотестов.
4
Генерация плана дня
Срочно — просрочки, 0 выдач, договор >7д, фичи без бизнес-цели
Внимание — изменения статусов, задачи без сроков в Tracker
Действия — готовые сообщения партнёрам (с @тегами контактных лиц)
Автотесты — фичи готовые к тесту, результаты прошлых тестов
Таблица — пустые колонки, несинхронизированные бэклоги
5
Предложение действий (Delivery Manager подтверждает)
Ничего не отправляется без подтверждения DM. Формат предложений:

7 Синхронизация бэклогов

Внутренний бэклог → партнёрские бэклоги. Автоматическое обновление при смене статуса.

Как работает синхронизация

1
Внутренний бэклог (источник правды)
Spreadsheet {SPREADSHEET_ID} — листы Партнеры, Фичи, Интеграции. Все изменения сначала тут.
2
/morning обнаруживает изменение статуса
Кеш: вчера статус "Создание верстки", сегодня "Done" → триггер синхронизации.
3
Проверка: есть ли у партнёра свой бэклог?
В Sheets у партнёра есть колонка "Бэклог ID" — если заполнена, это Spreadsheet ID партнёрского бэклога.
4
Обновление партнёрского бэклога
Клод предлагает: "Обновить бэклог Партнёра: статус → Done, лендинг готов, ждём дату запуска?"
Delivery Manager: "да" → Sheets: gsheets_update_cell в партнёрском Sheet.
Sheets: обновить строку в партнёрском бэклоге
5
Уведомление в Telegram
После обновления бэклога - предложить написать в чат партнёра о смене статуса.
Telegram: "@{контакт}, обновили статус: лендинг готов, ждём вашего подтверждения по дате запуска."

Что синхронизируется

Поле внутреннего бэклога→ Поле партнёрского бэклогаSLA обновления
Внутр. статусСтатус подготовкиВ тот же день
Внеш. статусСтатус согласованияВ тот же день
ДоговорСтатус договораВ тот же день
Ход работыКомментарий/прогрессПри изменении
Срок запускаПлановая датаПри согласовании
ПроверенТестирование пройденоПри прохождении
Правило: внутренний бэклог = источник правды. Партнёрский бэклог обновляется ИЗ внутреннего, не наоборот. Если партнёр обновил что-то в своём бэклоге - сначала перенести во внутренний.

8 Автотесты

Интеграция с Selenium-тестами команды. Клод подсвечивает фичи, готовые к тесту, и предлагает DM запуск.

Когда предлагаются автотесты

1
Триггер: статус в Tracker = "Готово к тесту" В тот же день
Клод в /morning сверяет Tracker через search_issues. Если фича перешла в "Готово к тесту" — подсвечивает и предлагает DM запустить тесты.
Tracker: search_issues(filter: {status: "readyForTest"})
2
Определение типа тестов
По тегам/компоненту задачи в Tracker:
Frontend → Selenium: формы, точки входа, редиректы, отображение витрины
Backend → API-тесты: постбеки, статусы заявок, расчёт комиссии, сплит
Партнёр → E2E: загрузка лендинга, подача заявки, проверка в АРМе
3
Запуск тестов Результат за 1 час
Selenium-скрипты подготовлены командой. Клод запускает по пути к тест-сьюту из конфига задачи.
Selenium: запуск тест-сьюта
4
Результат → Tracker + Telegram Сразу после тестов
Все тесты прошли → Tracker: комментарий "Автотесты пройдены" + move_issue → "Тестирование"
Есть падения → Tracker: комментарий со списком падений. Telegram: уведомление DM с деталями.
Tracker: add comment + move_issue Telegram: результат DM

Типы автотестов

ТипЧто проверяетИнструментSLA результата
FrontendФормы, точки входа, витрина, редиректы, мобильная версияSelenium30 мин
BackendAPI-ответы, постбеки МФО, расчёт комиссии, статусы заявокAPI-тесты15 мин
E2E партнёрЛендинг → заявка → статус в АРМе → постбек → отчётSelenium + API1 час
МониторингДоступность лендингов, время отклика, ошибки 5xxHTTP-проверки5 мин
Тест-сьюты готовит команда разработки. Клод подсвечивает где нужны тесты и предлагает DM их запустить. Путь к тест-сьюту хранится в конфиге задачи в Tracker (поле "Тесты" или тег).
